Driving GPU based particles using 2D textures, Unity

Millions of particles for the cost of a single render pass

May 5, 2024   Â·   3 min read

Table of contents

Objectives

  • to implement a light-weighted particle system using the Unity built-in render pipeline, fully instanced on the GPU, easy to use and compatible with a maximum amount of targets (such as Nintendo Switch or WebGL)

  • to have millions of particles following a hand-drawn path, reacting to gravity, and colliding with obstacles

  • to make the system efficient enough to be heavily instantiated in a scene

Solutions

  • Use of multiple data fields packed into a single 2D texture, such as normals, distances, elevations (…). The data is generated in screen space (XY, NDC), and linked to a 3D scene through a 3D object that fits the 2D space (screen mesh, quadlike). The 3D object is used as an handle to facilitate manipulations in the scene.

  • Use of a fixed size vector4 array to hold the values on the CPU (with proper accessors). The array is passed to the GPU in one block, as a constant buffer for the uniforms (method called aggressive packing). This way, no extra copies are made and we limit CPU to GPU calls, without using a compute buffer. However, this solution implies to construct custom interfaces to access the array’s elements, both from script and inspector.

Following a path

To drive particles along a path, we need :

  • a signed distance field to the outlines of the path
  • normals of the path outlines (for collisions and constaints)
  • a path direction (also called flow)
  • elevations (or heights) to raise the system to a third dimension, following a relief.

To see how SDF and directions can be computed from 2D drawings, check the post sdfGenerators

A 4 component’s texture (float or half) can pack everything, such as two positive values (packed in range [0,1]) set the length of 2 unit vectors (2D), normals and flow directions.

This kind of texture doesn’t need to be more defined than 512x512 pixels. Since we now have the distance to the path outlines and its normals, we can have the particle’s velocity vector reflecting around the normal if the distance is close to zero.

Gravity

To compute gravity particles, collisions with a topography, we pre-compute normals from an height map. The data textures encapsulate at least theses 3D normals and the corresponding height map.

By transforming the particle’s position to field space we can compare its Z component with the height value (or elevation, from the data texture), if less, we can once again reflect the particle velocity around the normal.
