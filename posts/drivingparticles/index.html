Resume

With GPU instanced particles, 2D textures (or Arrays of textures) can be used to provide a simulation shader (compute or fragment) the necessary data to implement behaviours and awarness of the scene.

To orient this data (normals, directions…) in the scene, the texture can be linked to a quad and to its matrix transform (relative to the domain of the particle system).

Packing as much data as possible in a single texture is key here, for optimisation purposes but also for readability and usability.

Path following

To drive particles along a path (shape), I use a signed distance field to the outlines of the path, normals of the path for collisions and constaints, a path direction vector and finally an height map to raise the system to 3d and have it following a relief.
To see how I compute SDF and directions from 2D drawings, check the post sdfGenerators

A 4 component’s texture (float or half) can pack everything, such as :

float4 col = _PathTex[uv * dim] // hlsl direct access
struct PathData
{
    // red + green
    float3 shapeNormal;         // float3(normalize(col.rg), 0)
    float distanceToShape;      // length(col.rg) * 2.0 - 1.0

    // blue + alpha
    float3 pathDirection;       // float3(normalize(col.ba), 0)
    float height;               // length(col.ba) * 2.0 - 1.0
}

// to get the vectors oriented in scene 
shapeNormal = mul(FieldToDomainMatrix, float4(shapeNormal,0)).xyz;

Gravity

float4 col = _TopoTex[uv * dim] // hlsl direct access
struct TopoData
{
    // red + green + blue
    float3 normals;             // normalize(col.rgb)
    float height;               // length(col.rgb) * 2.0 - 1.0

    // alpha
    float zone;                 // col.a
}
