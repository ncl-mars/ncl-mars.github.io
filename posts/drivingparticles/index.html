Driving GPU based particles using 2D textures, Unity

Millions of particles for the cost of a single render pass

Objectives

  • to implement a light-weighted particle system using the Unity built-in render pipeline, fully instanced on the GPU, easy to use and compatible with a maximum amount of targets (such as Nintendo Switch or WebGL)

  • to have millions of particles following a hand-drawn path, reacting to gravity, and colliding with obstacles

  • to make the system efficient enough to be instantiable per dozens in a scene

  • to make all of this running without the convenience neither of a compute shader nor a compute buffer (for compatibility purposes)

Solutions

Use of multiple data fields at once into a single 2D texture buffer, such as normals, distances, directions, elevations (…) and using a quad to import it into a scene, as binding the fields to a transform component;

Use of the particle’s position in field space to read that data, and using the decoded data (tranformed back to world space) as an input for a physic system (steering behaviours, reynolds rules…)

Use of a constant uniform buffer packed into a vector4 array (aggressive packing) to limit CPU > GPU calls

// C# pseudo code for computing matrices
void ComputeMatrices(Bounds domain, Transform quadTransform)
{
    Matrix4x4 FieldToDomain = Matrix4x4.TRS(
        domain.origin,                  // world position
        Quaternion.identity,            // bypass rotation 
        MinComponent(domain.size)       // preserve ratio
    ).inverse;
    
    FieldToDomain *= quadTransform.localToWorldMatrix;
    DomainToField  = FieldToDomain.inverse;
}

Following a path

To drive particles along a path (shape), we need :

  • a signed distance field (to the outlines of the path)
  • normals of the path outlines (for collisions and constaints)
  • a path direction (also called flow)
  • elevation (or height) to raise the system to a third dimension, following a relief.

To see how SDF and directions can be computed from 2D drawings, check the post sdfGenerators

A 4 component’s texture (float or half) can pack everything, such as :

// HLSL pseudo code
float4 col = _PathTex[uv * dim]
struct PathData
{
    // red + green
    float3 shapeNormal;         // float3(normalize(col.rg), 0)
    float distanceToShape;      // length(col.rg) * 2.0 - 1.0

    // blue + alpha
    float3 pathDirection;       // float3(normalize(col.ba), 0)
    float height;               // length(col.ba) * 2.0 - 1.0
}
// to transform from field to bounds space  
shapeNormal = mul(FieldToDomain, float4(shapeNormal,0)).xyz;

This kind of texture doesn’t need to be more defined than 512x512 pixels. Since we now have the distance to the path outlines and its normals, we ca have the particle’s velocity vector reflecting around the normal if the distance is close to zero

Gravity

To compute gravity and collision with topography, we pre-compute normals from an height map. Data textures for the gravity encapsulate at least 3D normals and the height map, such as :

// HLSL pseudo code
float4 col = _TopoTex[uv * dim] // hlsl direct access
struct TopoData
{
    // red + green + blue
    float3 normals;             // normalize(col.rgb)
    float height;               // length(col.rgb) * 2.0 - 1.0

    // alpha
    float zone;                 // col.a, optional
}

By converting the particle’s position to field space (by using the DomainToField matrix) we can compare its Z component with the height value (from the data texture), if less, we can once again reflect the particle velocity around
